# REVIEW: разбор кодовой базы `tpo_lab_2`

## 1) Что это за проект
Проект реализует вычисление набора унарных математических функций (тригонометрических и логарифмических), их объединение в составную `SystemFunction`, запуск расчётов по входному CSV и логирование каждого вызова модулей.

Ключевая идея: 
- вход: список `x` из CSV,
- обработка: выбранный модуль `ModuleType` вычисляет `f(x)`,
- выход: CSV с колонками `x,result`,
- аудит: лог-файл с записью каждого вычисления.

## 2) Архитектура по пакетам

### `org.savadanko.math` (точка входа)
- `App`:
  - CLI-роутер приложения;
  - поддерживает 2 режима:
    - `samples` — генерирует демонстрационные CSV и сразу прогоняет расчёты;
    - `<module> <input> <output> [log]` — исполняет конкретный модуль.
  - задаёт общие параметры точности:
    - `EPS = 1e-10`,
    - `MAX_ITERATIONS = 100_000`.

### `org.savadanko.math.functions` (математика)
- `UnaryFunction` — единый контракт: `double apply(double x)`.
- Базовые модули:
  - `SinFunction`, `CosFunction` — ряды Тейлора с нормализацией угла.
  - `LnFunction` — численное вычисление `ln(x)` через ряд для `z=(m-1)/(m+1)` + декомпозиция числа по степени двойки.
- Производные модули:
  - `TanFunction = sin/cos` через безопасное деление,
  - `CotFunction = 1/tan`,
  - `SecFunction = 1/cos`,
  - `LogFunction = ln(x)/ln(base)`.
- Составной модуль:
  - `SystemFunction` — piecewise-функция:
    - при `x <= 0` используется тригонометрическая ветка,
    - при `x > 0` используется логарифмическая ветка.

### `org.savadanko.math.utils` (склейка и утилиты)
- `FunctionFactory`:
  - собирает граф зависимостей функций;
  - оборачивает каждую функцию в логирующий декоратор `LoggingUnaryFunction`;
  - отдаёт функцию по `ModuleType`.
- `ModuleType`:
  - enum модулей + алиасы (`ctg` для `cot`, `log_2` и т.д.).
- `MathUtils`:
  - `normalizeAngle`, `safeDiv`, `isZero`, `powInt`.
- `MathConstants`:
  - `PI`, `TWO_PI`, `LN_2`.

### `org.savadanko.math.csv` (ввод/вывод)
- `CsvReader`:
  - читает CSV/semicolon-separated строки,
  - пропускает пустые строки, комментарии `#`, заголовок `x`,
  - берёт первую колонку и парсит `double`.
- `CsvWriter`:
  - пишет:
    - входные таблицы (`x`),
    - результаты (`x,result`),
  - корректно сериализует `NaN`, `Infinity`, `-Infinity`.
- `CsvFunctionExecutor`:
  - orchestration: `module + input.csv -> output.csv`.
- `SampleCsvGenerator`:
  - генерирует тестовые входы (`cos`, `ctg`, `system`) и сразу считает выходы.

### `org.savadanko.math.logger` (логирование)
- `AppLogger`:
  - создаёт файловый `java.util.logging.Logger`,
  - формат: `timestamp | level | message`.
- `LoggingUnaryFunction`:
  - декоратор над `UnaryFunction`,
  - логирует `module`, `input`, `result` для каждого вызова,
  - при исключении пишет `SEVERE` и пробрасывает ошибку дальше.

## 3) Поток выполнения (end-to-end)

1. Пользователь запускает `App` с аргументами.
2. `App` парсит модуль (`ModuleType.fromString`) и пути файлов.
3. `AppLogger.configure` поднимает логгер.
4. `FunctionFactory` строит все функции и зависимости.
5. `CsvFunctionExecutor.execute`:
   - читает `x` через `CsvReader`,
   - применяет выбранную функцию к каждому `x`,
   - пишет результат через `CsvWriter`.
6. Каждый `apply(x)` логируется декоратором `LoggingUnaryFunction`.

## 4) Математическая часть: как именно считается

### `SinFunction` и `CosFunction`
- Проверяют невалидные аргументы (`NaN`, `Infinity`) -> `NaN`.
- Нормализуют угол в диапазон около `[-pi, pi]` (`MathUtils.normalizeAngle`), что ускоряет сходимость ряда.
- Используют рекуррентное обновление члена ряда (без пересчёта факториалов), останавливаются по `|term| < eps` или по лимиту итераций.

### `Tan/Cot/Sec`
- Не считают напрямую, а строятся из базовых функций через `safeDiv`.
- Вблизи полюсов деление на почти ноль даёт `NaN` (через epsilon-порог).

### `LnFunction`
- Доменные случаи:
  - `x < 0` или `NaN` -> `NaN`,
  - `x = 0` -> `-Infinity`,
  - `x = +Infinity` -> `+Infinity`,
  - `x = 1` -> `0`.
- Для остальных значений:
  - раскладывает `x = mantissa * 2^k`,
  - считает `ln(mantissa)` рядом для `z=(m-1)/(m+1)`,
  - добавляет `k * ln(2)`.
- Это существенно устойчивее, чем прямой ряд по `(x-1)` для больших/малых `x`.

### `LogFunction`
- Предвычисляет `ln(base)` в конструкторе.
- Валидирует основание: если `ln(base)` невалидно или близко к нулю, бросает `IllegalArgumentException`.
- Затем использует `ln(x)/ln(base)`.

### `SystemFunction`
Функция кусочная:
- `x <= 0`: триг-ветка на `cot, tan, sec`.
- `x > 0`: лог-ветка на `log2, log3, log5, log10`.

Важный нюанс лог-ветки:
- в выражении есть `log2/log2` (по сути 1, но при `x=1` это `0/0 -> NaN`),
- также `(log2 - log2) * log2` всегда 0.

Итог: при `x=1` функция ожидаемо возвращает `NaN`, что покрыто тестами. Это похоже на лабораторное выражение, где сохранена исходная символическая форма.

## 5) Универсальный ввод/вывод CSV

Проект действительно делает ввод «универсальным» в рамках простого формата:
- принимает `,` и `;` как разделитель (за счёт split по regex `;|,`),
- читает только первый столбец,
- допускает комментарии и пустые строки.

Ограничения текущей реализации:
- нет обработки локали (десятичная запятая не поддерживается как часть числа),
- ошибки парсинга (`NumberFormatException`) не перехватываются в `CsvReader` для построчного отчёта,
- нет валидации наличия файла/прав доступа на уровне UX-сообщений (исключения уходят наружу).

## 6) Логирование результатов

Логирование реализовано чисто через декоратор и потому автоматически применяется к любому модулю, созданному фабрикой.

Что логируется:
- имя модуля,
- входное значение,
- результат.

Плюсы:
- единообразный формат,
- легко анализировать трассу вычислений,
- логи есть и для внутренних подмодулей `system`, что даёт прозрачность.

Риск:
- для больших CSV лог может быть очень объёмным (каждый вызов каждой вложенной функции в `system`).

## 7) Интеграционные тесты: что покрыто

Пакет `src/test/java/org/savadanko/math/functions` покрывает три уровня:

1. Интеграция между математическими модулями:
- `TanFunctionIntegrationTest` — `tan` через реальные `sin+cos`.
- `LogFunctionIntegrationTest` — `log_b` через реальный `ln`.

2. Интеграция веток `SystemFunction` с изоляцией противоположной ветки:
- `SystemFunctionTrigIntegrationTest`:
  - тригонометрическая ветка реальная,
  - логарифмические зависимости замоканы,
  - проверяется отсутствие вызовов лог-модулей при `x <= 0`.
- `SystemFunctionLogIntegrationTest`:
  - логарифмическая ветка реальная,
  - тригонометрические зависимости замоканы,
  - проверяется отсутствие вызовов trig-модулей при `x > 0`.

3. Сквозная проверка:
- `SystemFunctionE2ETest` сравнивает полный `SystemFunction` с эталонной реализацией на `java.lang.Math`.
- `CsvFunctionExecutorIntegrationTest` проверяет pipeline CSV -> вычисление -> CSV + факт логирования.

Это хорошее покрытие именно интеграционного уровня: проверяются не только формулы, но и связь модулей, маршрутизация веток и IO-контур.

## 8) Качество дизайна: сильные стороны

- Чёткое разделение ответственности по пакетам.
- Единый интерфейс `UnaryFunction` позволяет легко комбинировать модули.
- `FunctionFactory` централизует граф зависимостей.
- Декоратор логирования не загрязняет математические классы.
- Численные алгоритмы (`sin`, `cos`, `ln`) реализованы аккуратно и устойчиво для учебной задачи.
- Интеграционные тесты действительно проверяют взаимодействие, а не только unit-детали.

## 9) Замечания и точки роста

1. `FunctionFactory#create` на каждый вызов строит весь граф функций заново.
- Для batch-запуска это обычно нормально,
- но при частых вызовах можно кэшировать инстансы.

2. `MathUtils` помечен `@NoArgsConstructor` и одновременно `final` с `static`-методами.
- Lombok тут не нужен; достаточно приватного конструктора или вообще убрать аннотацию.

3. В `SystemFunction.logBranch` есть алгебраически избыточные части (`log2/log2`, `(log2-log2)*log2`).
- Вероятно, это нужно для соответствия постановке лабораторной,
- но для production-версии формулу стоило бы упростить и явно задокументировать особые точки.

4. CSV-обработка минималистична.
- Нет rich-диагностики ошибок по строкам,
- нет поддержки quoted values/сложного CSV (если это потребуется).

5. Конфигурируемость.
- `EPS` и `MAX_ITERATIONS` захардкожены в `App`; удобно было бы вынести в CLI-флаги или конфиг.

## 10) Почему проект соответствует вашему описанию

- Поддержка тригонометрических и логарифмических функций: реализована отдельными модулями + `SystemFunction`.
- Интеграционное тестирование: реализовано набором integration/e2e тестов с реальными зависимостями и моками на границах веток.
- Универсальный ввод и логирование результатов: есть CSV-конвейер (`CsvReader/Writer/Executor`) и сквозное логирование вызовов через `LoggingUnaryFunction`.
